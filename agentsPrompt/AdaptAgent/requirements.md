# Platform Adaptation Module Requirements (平台适配模块需求)

## 1. 平台发布记录表设计 (platform_publish_record)
```sql
CREATE TABLE platform_publish_record (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    task_id BIGINT NOT NULL,
    platform VARCHAR(50) NOT NULL,                -- xiaohongshu, zhihu, wechat, bilibili
    adapted_title VARCHAR(255),                   -- 适配后标题
    adapted_content CLOB,                         -- 适配后正文
    tags_json CLOB,                               -- 适配后标签数组 JSON
    adapted_media_json CLOB,                      -- 适配后的插图顺序与封面引用 JSON
    publish_mode VARCHAR(50) NOT NULL,            -- MOCK, REAL
    status VARCHAR(50) NOT NULL,                  -- PENDING, ADAPTING, READY, PUBLISHING, SUCCESS, FAILED 等
    publish_url VARCHAR(500),                     -- 模拟或真实的发布链接
    error_message VARCHAR(1000),                  -- 错误堆栈/失败说明
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    published_at TIMESTAMP,
    FOREIGN KEY (task_id) REFERENCES content_task(id)
);
```

## 2. 核心接口定义
设计统一的适配层，解耦核心分发逻辑与各大模型/平台专有逻辑：
```java
public interface PlatformAdapter {
    String platform(); // 获取当前适配的平台名称，如 "xiaohongshu"
    AdaptedContent adapt(NormalizedContent content, PlatformRule rule); // 执行转换适配
    ValidationResult validate(AdaptedContent content, PlatformRule rule); // 针对字数/格式规则校验
}
```

## 3. 平台特色风格定义
*   **微信公众号 (wechat)**：适合长文深度阅读，采用清晰排版，使用 HTML 正文保留样式，有摘要。
*   **知乎 (zhihu)**：采用 Markdown 结构排版，风格专业、论证理性严谨，配图合理，有参考链接。
*   **小红书 (xiaohongshu)**：字数小于 1000，表达口语化、种草风，有大量自然 Emoji 穿插，文末带多个话题标签（如 `#XX`）。
*   **B站 (bilibili)**：专栏长文或动态短文格式，适合粉丝引流。

## 4. 异步处理与降级机制
*   **异步机制**：前端触发适配后，同步将 `platform_publish_record` 表中的状态置为 `ADAPTING`，并在业务层生成唯一的适配 `recordId` 队列返回。大模型的网络交互放到后台线程池中，每个平台计算完成后独立落库并进行 WS 推送，保证工作台的“生成流水线效果”。
*   **兜底降级**：若检测到 API Key 未配置，或发生请求连接超时（5秒），需立刻转由 `TemplatePlatformAdapter` 采用内置的规则模板套用转换生成静态演示内容。此时：
    *   在 `platform_publish_record` 记录中正常生成内容，状态设为 `READY`。
    *   通过 WebSocket 发送 `PLATFORM_ADAPT_DEGRADED` 事件，告知前端已进行降级处理。
