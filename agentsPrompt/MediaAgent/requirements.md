# Media Resource Module Requirements (媒体资源模块需求)

## 1. 媒体资源表设计 (media_resource)
```sql
CREATE TABLE media_resource (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    task_id BIGINT NOT NULL,
    original_name VARCHAR(255) NOT NULL,
    mime_type VARCHAR(100) NOT NULL,
    size_bytes BIGINT NOT NULL,
    width INT,
    height INT,
    storage_type VARCHAR(50) NOT NULL,           -- LOCAL 等
    storage_key VARCHAR(500) NOT NULL,           -- 本地磁盘路径或 OSS 键
    public_url VARCHAR(500) NOT NULL,            -- Web 可访问的地址
    sha256 VARCHAR(128),                          -- 文件哈希校验值
    status VARCHAR(50) NOT NULL,                  -- UPLOADED, READY, FAILED 等
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (task_id) REFERENCES content_task(id)
);
```

## 2. 文件存储与访问规则
*   **物理存储路径**：本地开发环境保存在项目目录下的 `data/media/{taskId}/{sha256}.{ext}`，按任务（`taskId`）进行分目录管理。
*   **公共访问 URL**：图片公共 URL 格式为 `http://localhost:8080/media/{mediaId}`。该路由不能带 `/api` 前缀，以方便前端页面、浏览器插件和拟态详情页直接发起二进制请求。
*   **去重和缓存**：通过计算文件的 SHA-256 对重复图片进行检验，避免重复存储。

## 3. 强一致性安全删除机制
为了防止内容任务的正文或发布记录产生死链，删除媒体资源（`DELETE /api/tasks/{taskId}/media/{mediaId}`）时必须执行严格的引用校验：
1.  **第一步：检验所属关系**：确认待删除的媒体资源记录 `media_resource` 的 `task_id` 确实与请求路径中的 `taskId` 一致。
2.  **第二步：检索标准内容引用**：读取 `content_task` 的 `normalized_content_json` 字段，解析该 JSON 并检索是否包含 `mediaId`。
3.  **第三步：检索发布记录引用**：读取 `platform_publish_record` 列表中所有关于该任务的记录，解析 `adapted_media_json` 字段确认是否包含该 `mediaId`。
4.  **校验冲突处理**：如果发现任何引用存在，**必须拒绝删除**。返回业务状态码 `409`，并告知前端具体的冲突点（例如由 `normalizedContent` 或具体的发布记录 ID 引用），提醒用户先移出引用。
