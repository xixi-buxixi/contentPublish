# Pulse Distro Java 轻量化实践方案

## Java 21 + Spring Boot 3.3 + LangChain4j 单体实现说明

| 版本 | 修订日期 | 状态 | 作者 | 变更说明 |
| ---- | -------- | ---- | ---- | -------- |
| v1.3.0 | 2026-05-29 | 优化稿 | 独立开发者 | 收敛为纯 Java MVP，补充媒体资源、异步适配、Mock 页面、插件在线态和配置驱动扩展 |

## 1. 可行性结论

该 Java 轻量化方案是可行的，适合作为课程项目、答辩演示或小型个人实践。相比“Java + Python + Redis + MySQL + Chrome 插件”的完整架构，它把运行环境收敛到一个 Spring Boot Jar 中，降低了部署复杂度，也更容易把题目要求演示完整。

推荐的 MVP 边界如下：

1. 必做：内容输入、图片上传、本地媒体存储、四个平台内容适配、多端预览、人工编辑、一键模拟发布、Mock 详情页。
2. 建议做：LangChain4j 接入真实大模型；如果没有 API Key，则提供规则模板降级。
3. 可选做：Chrome 插件真实发布、插件心跳、验证码人工接管。

核心判断：**模拟发布是主线，真实发布是扩展**。这样既符合题目“一键发布或模拟发布”的要求，又不会被平台风控和验证码拖住项目交付。

## 2. 轻量化架构

系统运行在单个 JVM 进程中，默认不依赖独立 Redis、MySQL 或 Python 服务。

```text
┌────────────────────────────────────────────────────────────────┐
│ 单体 Spring Boot Jar                                            │
│                                                                │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ Vue 3 静态前端                                            │  │
│  │ 工作台 / 多平台预览 / Mock 页面 / WebSocket 流水线          │  │
│  └──────────────────────────▲───────────────────────────────┘  │
│                             │ HTTP + WebSocket                  │
│                             ▼                                  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ Spring Boot 3.3 核心服务                                  │  │
│  │ TaskService / MediaService / AdaptService / PublishService│  │
│  │ PlatformConfigService / PluginManager                     │  │
│  └───────────────┬───────────────────────────┬──────────────┘  │
│                  │                           │                 │
│                  ▼                           ▼                 │
│  ┌─────────────────────────────┐ ┌──────────────────────────┐  │
│  │ LangChain4j / 模板降级适配器 │ │ MockPublisher 默认发布器  │  │
│  │ 平台风格改写 / 标题 / 标签   │ │ 本地平台拟态详情页         │  │
│  └─────────────────────────────┘ └──────────────────────────┘  │
│                  │                           │                 │
│                  ▼                           ▼                 │
│  ┌─────────────────────────────┐ ┌──────────────────────────┐  │
│  │ H2 本地数据库                │ │ 本地媒体文件存储           │  │
│  │ task / record / media / cfg  │ │ data/media/{taskId}/...    │  │
│  └─────────────────────────────┘ └──────────────────────────┘  │
└────────────────────────────────────────────────────────────────┘
```

设计原则：

1. 单 Jar 启动，减少环境问题。
2. H2 使用文件模式，避免每次重启丢数据。
3. 图片保存到本地目录，并通过 `/media/{id}` 暴露为浏览器可访问 URL。
4. 平台适配异步执行，结果通过 WebSocket 增量推送。
5. 平台规则配置化，避免每新增平台都改核心流程。

## 4. 标准内容模型

文档解析后的内容统一进入标准模型。图片只保存媒体 ID 和可访问 URL，不保留 `local://image-1.png` 这类临时路径。

```java
public record MediaRef(
    Long mediaId,
    String publicUrl,
    String alt,
    Integer width,
    Integer height
) {}

public record ContentBlock(
    String type,       // heading, paragraph, list, image
    Integer level,
    String text,
    MediaRef media
) {}

public record NormalizedContent(
    String title,
    String summary,
    List<ContentBlock> blocks
) {}

public record AdaptedContent(
    String platform,
    String title,
    String content,
    List<String> tags,
    List<MediaRef> media,
    String styleExplanation
) {}
```

## 5. 数据模型设计

### 5.1 内容任务表

```sql
CREATE TABLE content_task (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    source_type VARCHAR(50) NOT NULL,
    raw_content CLOB NOT NULL,
    normalized_content_json CLOB,
    cover_media_id BIGINT,
    status VARCHAR(50) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

说明：`updated_at` 建议由 JPA 的 `@UpdateTimestamp` 或业务代码更新，不依赖 MySQL 风格的 `ON UPDATE` 语法，避免 H2 兼容性问题。

### 5.2 媒体资源表

```sql
CREATE TABLE media_resource (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    task_id BIGINT NOT NULL,
    original_name VARCHAR(255) NOT NULL,
    mime_type VARCHAR(100) NOT NULL,
    size_bytes BIGINT NOT NULL,
    width INT,
    height INT,
    storage_type VARCHAR(50) NOT NULL,
    storage_key VARCHAR(500) NOT NULL,
    public_url VARCHAR(500) NOT NULL,
    sha256 VARCHAR(128),
    status VARCHAR(50) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (task_id) REFERENCES content_task(id)
);
```

本地实践建议：

1. 图片保存到 `data/media/{taskId}/{sha256}.{ext}`。
2. `public_url` 存为 `http://localhost:8080/media/{id}`。
3. Mock 页面直接用 `public_url` 展示图片。
4. 真实发布插件通过 `fetch(publicUrl)` 下载为 `Blob/File` 后再上传到平台页面。

### 5.3 平台发布记录表

```sql
CREATE TABLE platform_publish_record (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    task_id BIGINT NOT NULL,
    platform VARCHAR(50) NOT NULL,
    adapted_title VARCHAR(255),
    adapted_content CLOB,
    tags_json CLOB,
    adapted_media_json CLOB,
    publish_mode VARCHAR(50) NOT NULL,
    status VARCHAR(50) NOT NULL,
    publish_url VARCHAR(500),
    error_message VARCHAR(1000),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    published_at TIMESTAMP,
    FOREIGN KEY (task_id) REFERENCES content_task(id)
);
```

### 5.4 平台配置表

```sql
CREATE TABLE platform_config (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    platform VARCHAR(50) NOT NULL UNIQUE,
    display_name VARCHAR(100) NOT NULL,
    config_json CLOB NOT NULL,
    enabled BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 5.5 插件会话表（真实发布可选）

```sql
CREATE TABLE plugin_session (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id BIGINT NOT NULL,
    session_id VARCHAR(100) NOT NULL,
    extension_version VARCHAR(50),
    browser VARCHAR(100),
    status VARCHAR(50) NOT NULL,
    last_heartbeat_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

MVP 如果只做模拟发布，可以先不建插件表；但文档保留该表有利于说明未来真实发布的工程闭环。

## 6. 核心业务流程

### 6.1 创建任务与上传媒体

推荐流程：

1. 前端先调用 `/api/tasks` 创建内容任务。
2. 若有图片，调用 `/api/tasks/{taskId}/media` 上传图片。
3. 后端保存图片，写入 `media_resource`，返回 `mediaId` 和 `publicUrl`。
4. 前端把图片插入正文时使用 `mediaId`，提交时由后端生成 `NormalizedContent`。

图片上传接口：

```http
POST /api/tasks/{taskId}/media
Content-Type: multipart/form-data
```

响应：

```json
{
  "mediaId": 501,
  "publicUrl": "http://localhost:8080/media/501",
  "mimeType": "image/png",
  "width": 1200,
  "height": 800
}
```

### 6.2 异步平台适配

LLM 适配四个平台可能需要 10 到 15 秒，不建议使用同步 HTTP 阻塞等待。

```http
POST /api/tasks/{taskId}/adapt
Content-Type: application/json
```

```json
{
  "platforms": ["xiaohongshu", "zhihu", "wechat", "bilibili"]
}
```

响应：

```json
{
  "taskId": 1001,
  "status": "PROCESSING",
  "message": "平台适配已开始，请监听 WebSocket。"
}
```

后台使用 `ThreadPoolTaskExecutor` 或 `ExecutorService` 并行处理，每个平台完成后推送：

```json
{
  "event": "PLATFORM_ADAPT_COMPLETED",
  "data": {
    "taskId": 1001,
    "recordId": 9001,
    "platform": "xiaohongshu",
    "status": "READY",
    "title": "别再手动复制了，内容分发这样做更快",
    "media": [
      {
        "mediaId": 501,
        "publicUrl": "http://localhost:8080/media/501"
      }
    ]
  }
}
```

### 6.3 人工预览与编辑

前端以平台 Tab 展示适配结果。用户修改后的标题、正文、标签、图片顺序需要回写到 `platform_publish_record`，发布时以人工确认后的版本为准。

### 6.4 一键模拟发布

```http
POST /api/tasks/{taskId}/publish
Content-Type: application/json
```

```json
{
  "mode": "mock",
  "platforms": ["xiaohongshu", "zhihu", "wechat", "bilibili"]
}
```

`MockPublisher` 执行：

1. 校验标题、正文、标签、图片数量。
2. 校验 `media_resource.public_url` 可访问。
3. 将记录状态改为 `SUCCESS`。
4. 生成 `/mock/{platform}/{recordId}`。
5. 通过 WebSocket 推送发布成功。

### 6.5 Mock 页面表现力

Mock 页面是演示加分点，不应只是普通链接。建议实现四套轻量拟态模板：

| 平台 | Mock 页面表现 |
| ---- | ------------- |
| 小红书 | 手机壳预览、头像、图文卡片、Emoji 正文、话题标签、评论占位 |
| 知乎 | PC 问答/专栏布局、标题、作者信息、Markdown 正文 |
| 公众号 | 手机端长文排版、封面图、摘要、段落留白 |
| B站 | 专栏或动态样式、封面、互动按钮占位 |

这样可以直观证明“同一份内容被适配成不同平台风格”。

## 7. LangChain4j 与降级方案

推荐把 LLM 适配封装在统一接口中，而不是为每个平台写死一个方法。

```java
public interface PlatformAdapter {
    String platform();

    AdaptedContent adapt(NormalizedContent content, PlatformRule rule);

    ValidationResult validate(AdaptedContent content, PlatformRule rule);
}
```

`GenericLlmPlatformAdapter` 读取平台 JSON 中的 `stylePrompt`、字数限制、标签规则，组装 Prompt 后调用 LangChain4j。`TemplatePlatformAdapter` 作为无 API Key 时的降级实现，使用模板规则生成演示内容。

平台配置示例：

```json
{
  "platform": "xiaohongshu",
  "displayName": "小红书",
  "maxTitleLength": 20,
  "maxContentLength": 1000,
  "maxTags": 10,
  "supportsMarkdown": false,
  "image": {
    "maxCount": 9,
    "acceptedMimeTypes": ["image/jpeg", "image/png", "image/webp"]
  },
  "stylePrompt": "改写为口语化、轻量、适合种草分享的风格，保留核心观点，增加自然 Emoji 和话题标签。"
}
```

这种方式比在 `PlatformAgentMatrix` 中不断新增方法更适合小项目，因为扩展平台主要改配置，不会让核心接口持续膨胀。

## 8. Chrome 插件扩展（可选）

真实发布不是 MVP 主线，但可以保留接口设计。

插件激活时注册：

```http
POST /api/plugin/register
Content-Type: application/json
```

```json
{
  "userId": 1,
  "sessionId": "plugin-session-001",
  "extensionVersion": "1.0.0",
  "browser": "Chrome"
}
```

插件每 15 到 30 秒心跳一次。真实发布前，`PublishService` 先检查：

```java
if (mode == PublishMode.REAL && !pluginManager.isOnline(userId)) {
    throw new BusinessException("请先开启您的浏览器分发插件");
}
```

真实图片上传流程：

1. 后端把 `publicUrl` 下发给插件。
2. 插件 `fetch(publicUrl)` 获取图片。
3. 插件转换为 `Blob/File`。
4. 插件注入文件输入框或模拟拖拽上传。
5. 失败或验证码出现时，上报 `SUSPENDED`。

## 9. 实现优先级

### 第一阶段：稳定可演示

1. Spring Boot 单体项目。
2. H2 文件数据库。
3. 文本输入和图片上传。
4. 媒体资源表与 `/media/{id}` 访问。
5. 四个平台模板适配。
6. WebSocket 增量状态推送。
7. 一键模拟发布和四个平台 Mock 页面。

### 第二阶段：智能增强

1. 接入 LangChain4j。
2. 加入 LLM 失败后的模板降级。
3. 增加平台规则校验。
4. 支持 Markdown 到 HTML、Markdown 到纯文本转换。

### 第三阶段：真实发布探索

1. Chrome 插件注册和心跳。
2. 插件脚本自动填表。
3. 图片 URL 到 Blob/File 上传。
4. 验证码和人工接管。

## 10. 主要风险与优化建议

| 风险 | 影响 | 优化建议 |
| ---- | ---- | -------- |
| LLM 响应慢 | 前端等待时间长 | 适配接口异步化，WebSocket 增量推送 |
| 没有 API Key | 无法展示智能改写 | 提供模板降级适配器 |
| 图片路径不可访问 | Mock 页面和真实上传失败 | 所有图片先进入 `media_resource`，统一生成 `publicUrl` |
| H2 并发能力有限 | 多用户场景受限 | 项目演示可接受，生产环境替换 PostgreSQL/MySQL |
| 真实发布不稳定 | 受平台风控影响 | 默认模拟发布，真实发布作为可选扩展 |
| 平台扩展改核心代码 | 后期维护困难 | 配置驱动平台规则，必要时才新增专属 Adapter |

## 11. 总结

Java 轻量化方案适合作为本题的小型项目实践。它保留了题目需要的“输入、自动适配、一键发布、平台扩展”主链路，又通过 H2、本地媒体存储、MockPublisher 和配置驱动平台规则降低了工程复杂度。最终交付物可以是一个 Jar 包，启动后访问 `http://localhost:8080` 即可完成演示。
